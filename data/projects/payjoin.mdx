[Payjoin](https://payjoin.org/) is a protocol that lets the sender and
receiver of a Bitcoin transaction cooperate to build a single transaction
with inputs from both parties. The resulting transaction breaks the
[common-input-ownership heuristic](https://en.bitcoin.it/wiki/Privacy#Common-input-ownership_heuristic)
that chain analysis tools rely on, and at the same time batches two payments
into one, which saves on fees. Used widely, Payjoin makes the common-input
heuristic useless across the whole network, not only for the people using it.

The Payjoin Foundation is the non-profit stewarding this work, led by
[Dan Gould](https://github.com/DanGould). Its flagship output is the
[Payjoin Dev Kit](https://payjoindevkit.org/), a Rust implementation
([rust-payjoin](https://github.com/payjoin/rust-payjoin)) with bindings for
Python, Dart, Swift, and Kotlin. The dev kit powers [BIP
77](https://github.com/bitcoin/bips/pull/1483), Async Payjoin, which uses
[HPKE-encrypted Oblivious HTTP](https://ietf-wg-ohai.github.io/oblivious-http/draft-ietf-ohai-ohttp.html)
so sender and receiver can exchange messages without both being online at
once. That removed the biggest practical barrier to adoption: Payjoin v1
needed the receiver to run a public server, often behind Tor and TLS, which
very few wallets ever did.

## Why fund it?

Bitcoin privacy on-chain needs widespread adoption of tools that actively
break surveillance heuristics. A clean, production-ready library gives wallet
and service developers a drop-in way to add this behavior by default, without
each project building its own implementation.

OpenSats first funded this work as the Payjoin Dev Kit in the
[first wave of Bitcoin grants](/blog/bitcoin-grants-july-2023#payjoin-dev-kit)
in July 2023, and renewed the grant in the
[eleventh wave](/blog/eleventh-wave-of-bitcoin-grants) and again in the
[seventeenth wave](/blog/seventeenth-wave-of-bitcoin-grants) now directly to
the Payjoin Foundation. For a longer write-up, see the impact reports on
[Async Payjoin](/blog/developing-advancements-in-onchain-privacy#async-payjoin)
and the
[Payjoin Dev Kit](/blog/advancements-in-developer-libraries#payjoin-dev-kit).

## What's next?

[Bull Bitcoin](https://bullbitcoin.com/) shipped Payjoin v2 in December 2024.
[Cake Wallet](https://cakewallet.com/) followed with send and receive support
in May 2025. Integrations are in progress with
[Liana](https://github.com/wizardsardine/liana/),
[Galoy's Bria](https://github.com/GaloyMoney/bria),
[Boltz](https://boltz.exchange/), and
[LDK-Node](https://github.com/lightningdevkit/ldk-node) for Lightning channel
funding.

The `payjoin` crate has moved from version 0.11 to 0.24, adding a
session-event log so apps can show transaction history, a stable interface
for `payjoin-cli`, and the
[`ohttp-relay`](https://crates.io/crates/ohttp-relay) crate that
[Cake Wallet](https://cakewallet.com/) and [BOB Space](https://bobspaces.net/)
now run as public relays. Ongoing work focuses on multi-party Payjoin,
receiver cut-through batching, and bindings for more languages so any wallet
can adopt it without a Rust toolchain.
