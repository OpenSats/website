[Floresta](https://getfloresta.org/) is a Bitcoin full node written in Rust.
It keeps storage and memory requirements low by combining
[Utreexo](https://eprint.iacr.org/2019/611) (a cryptographic accumulator that
replaces the UTXO set with a small set of Merkle roots) with
[PoW Fraud Proofs](https://blog.dlsouza.lol/2023/09/28/pow-fraud-proof.html)
and pruning. The goal is to let people verify Bitcoin on hardware they already
own, including phones and small single-board computers, without giving up
trust-minimization.

The project ships two main pieces:
[`libfloresta`](https://github.com/getfloresta/Floresta/tree/master/crates),
a set of reusable crates that applications can embed, and
[`florestad`](https://github.com/getfloresta/Floresta/tree/master/bin/florestad),
a daemon that runs as a standalone node with a watch-only wallet and a built-in
Electrum server. Developers can use `libfloresta` the way they would use
[`rust-bitcoin`](https://github.com/rust-bitcoin/rust-bitcoin) or
[`bdk`](/projects/bdk), building wallets, explorers, and indexers on top of a
validating client. Script verification runs through the `libbitcoinkernel`
bindings so Floresta shares the same consensus rules as Bitcoin Core, and the
project has its own [mdBook documentation](https://getfloresta.github.io/floresta-docs/)
covering the codebase chapter by chapter.

## Why fund it?

Running a full node is the strongest form of trust-minimization on Bitcoin.
Resource requirements push most users onto custodial services or light clients
that trust third parties. Floresta does real validation with a footprint small
enough to run on commodity hardware, which makes it a good fit for phones,
single-board computers, and embedded wallet stacks.

OpenSats first funded Floresta in the
[seventh wave of Bitcoin grants](/blog/bitcoin-grants-september-2024-7th-wave#floresta)
in September 2024. The [fourteenth wave](/blog/fourteenth-wave-of-bitcoin-grants#floresta)
added grants for contributors
[João Leal](https://github.com/jaoleal),
[Moises Pompilio](https://github.com/moisesPompilio), and
[Jose Castillo López](https://github.com/JoseSK999). The project lives at
[getfloresta/Floresta](https://github.com/getfloresta/Floresta), with origins
at [Vinteum](https://vinteum.org/), the Brazilian nonprofit that incubated it.

## What's next?

Floresta [v0.9.0](https://github.com/getfloresta/Floresta/releases/tag/v0.9.0)
shipped in April 2026. On the wire, the node speaks BIP-0183 Utreexo messaging
for proof-driven sync. Script validation moved from the deprecated
`libbitcoinconsensus` to `libbitcoinkernel`. The JSON-RPC picked up
`rescanblockchain`, `gettxout`, `ping`, `addnode`, and JSON-RPC v1 for legacy
clients, so most Bitcoin Core tooling works against `florestad` unchanged.
v0.9.0 also landed a modular mempool (the groundwork for transaction relay),
parallel `pytest` functional tests, more fuzz targets, `no_std` portability
work, and dual licensing under MIT OR Apache-2.0. For reference,
[v0.8.0](https://github.com/getfloresta/Floresta/releases/tag/v0.8.0)
(July 2025) added BIP-0324 P2P v2, testnet4, Prometheus metrics, and the first
`mdBook`;
[v0.8.1](https://github.com/getfloresta/Floresta/releases/tag/v0.8.1)
(January 2026) was a bugfix release covering reindex and filter issues.

The next big item is
[SwiftSync](https://delvingbitcoin.org/t/swiftsync-speeding-up-ibd-with-pre-generated-hints/1562).
It pairs implicit deletion in the Utreexo accumulator with pre-generated hints
so blocks can be validated out of order without downloading proofs. In one
test, syncing signet to block 290,000 on a home connection went from 1h 15min
with plain Utreexo to 6 minutes with witnessless SwiftSync. The
[Utreexo BIP draft](https://github.com/bitcoin/bips/pull/1923) is under review.
Other open work: evaluating [BDK](/projects/bdk) as the watch-only wallet,
packaging `florestad` for [nix-bitcoin](https://nixbitcoin.org/) and upstream
[nixpkgs](https://github.com/NixOS/nixpkgs), transaction relay on top of the
new mempool, and more P2P integration tests.
